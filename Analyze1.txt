  The Fleet Manager — what it does every 5 minutes

  New file: server/src/services/autonomous-fleet-manager.service.ts

  every 5 minutes:
    1. for each running campaign:
         hourly_projection[campaign] = distribute(
           remaining_sends = Σ file.row_count × active_steps - stats_daily[today].sends,
           window = config.send_window_start..send_window_end in config.timezone,
           cap = config.daily_send_cap
         )
       → array of 24 buckets per campaign, in UTC

    2. aggregate across all campaigns:
         demand[h] = Σ_campaigns hourly_projection[campaign][h]
         for h in [now, now+24h]

    3. required_agents[h] = demand[h] / SENDS_PER_AGENT_PER_HOUR   (tunable, default ~180)
       required_containers[h] = ceil(required_agents[h] / AGENTS_PER_CONTAINER)

    4. target = max(required_containers[now..now+2h])  // look-ahead, avoid cold-start lag
       current = count(mcp_worker_instances where status='active')

       if target > current:   scale_up(target - current)          // immediate
       if target < current:   schedule drain(current - target)    // anti-flap: wait 15min

    5. write decision into system_settings.mcp.fleet.plan for audit + admin UI

    6. for scale actions, write access_logs event_type='mcp_autoscale'

  Why hourly (not daily) matters

  A user with send_window 09:00–17:00 America/Los_Angeles at 10k/day sends zero between 17:00–09:00 LA time. Aggregate 1,000 such users and the demand curve looks like:

  Sends/hour across fleet (1000 users × 10k/day, mixed US timezones):

    12M ─┐
        │       ╭──╮
     8M ─┤    ╭──╯  ╰─╮
        │   ╭──╯       ╰──╮
     4M ─┤  ╱             ╲
        │ ╱                ╲
     0M ─┴──┬──┬──┬──┬──┬──┬──┬──
          0  4  8 12 16 20  0  UTC

  Peak is ~3× average. If you size for average, peak work queues backlog; if you size for peak 24/7, you waste 60% of capacity overnight.

  With hourly autoscaling:
  - Night (UTC 00–12): 2 containers idle-watching
  - Peak (UTC 13–22): scale to 8 containers
  - Cost: ~50% of flat-peak-sizing, same SLA

  At 100k users this is $30k/mo savings vs static-peak fleet.
